# Golden Answer — How Does the NestJS CQRS Heroes Module Actually Work?

## intent_category
Code Onboarding & Comprehension

## golden_answer

When a hero kills a dragon and the application dispatches a KillDragonCommand, here is what happens from start to finish.

The `CommandBus` receives the command and looks up its handler using an internal map it built at application startup. That map was populated when NestJS scanned all providers decorated with `@CommandHandler`. The `@CommandHandler(KillDragonCommand)` decorator on `KillDragonHandler` attaches metadata to the class that tells the framework which command type this handler is responsible for. When `CqrsModule` initialises, it reads that metadata from every registered handler and stores the mapping. So by the time `execute()` is called, the bus already knows exactly which handler to invoke. `KillDragonCommand` itself needs no decorator — it is a plain class. The decorator lives on the handler, not the command.

`KillDragonHandler.execute()` receives the command and immediately retrieves the hero from `HeroRepository.findOneById()`. The repository returns a `Hero` instance, but at this point that instance is not yet wired to the `EventBus`. `AggregateRoot`, which `Hero` extends, provides `publish()` and `publishAll()` as methods, but their default implementations do nothing. They are intentional stubs. If you called `hero.commit()` at this point, your events would vanish silently. The line that fixes this is `this.publisher.mergeObjectContext(hero)`. `EventPublisher.mergeObjectContext()` reaches into the hero instance and replaces those stub methods with real implementations that route through the `EventBus`. From this point forward, any event that hero commits will actually be delivered.

With the hero now wired up, the handler calls `hero.killEnemy(dragonId)`. Inside `killEnemy()`, the `// logic` comment is a placeholder — this is where you would write code to update the hero's own in-memory state, such as incrementing a kill counter or removing the enemy from an active list. That state mutation is intentionally left as a placeholder. What comes after it is `this.apply(new HeroKilledDragonEvent(this.id, enemyId))`. Calling `apply()` does not dispatch the event anywhere. It pushes the event object onto an internal list inside `AggregateRoot` called `INTERNAL_EVENTS` and stops there. Nothing has reached the `EventBus` yet.

The handler then calls `hero.commit()`. This is the moment that actually triggers delivery. `commit()` calls `publishAll()`, which iterates over every event sitting in `INTERNAL_EVENTS` and calls `publish()` for each one. Because `mergeObjectContext()` already patched those methods, `publish()` now routes each event through the `EventBus`, where it becomes visible to every registered event handler and every active saga subscription.

`HeroesGameSagas` is already listening before any of this happens. When `CqrsModule` initialises at application startup, it scans all providers for properties decorated with `@Saga()`. When it finds `dragonKilled` on `HeroesGameSagas`, it calls `dragonKilled` as a function and passes in the `EventBus`'s observable stream as `events$`. That stream emits every event published anywhere in the application. `CqrsModule` then subscribes to the `Observable<ICommand>` that `dragonKilled` returns and keeps that subscription alive for the lifetime of the application. The saga itself does not manage its own subscription — the framework does.

Inside `dragonKilled`, `ofType(HeroKilledDragonEvent)` filters the stream so only events of that specific class pass through. `delay(1000)` holds each passing event for one second using RxJS stream timing, not a blocking sleep. After the delay, `map()` transforms the event into a `new DropAncientItemCommand`. The saga's job ends there. It returns the command inside the observable stream. `CqrsModule`, which subscribed to that stream, receives the emitted command and dispatches it through the `CommandBus` automatically. No manual dispatch code appears anywhere inside the saga.

`DropAncientItemHandler` picks up the command and runs through exactly the same cycle as `KillDragonHandler`: retrieve the hero, call `mergeObjectContext()`, call `hero.addItem(itemId)`, then `hero.commit()`. Yes, this is the universal pattern. Every domain operation in this codebase, whether triggered directly by a user request or indirectly by a saga, goes through the same apply-commit-publish cycle. The saga is just another source of commands — the handler does not know or care how the command arrived.

Regarding how `CqrsModule` reaches providers inside `HeroesGameModule` without appearing in that module's imports: `CqrsModule.forRoot()` in `AppModule` registers the module as globally scoped in NestJS. A global module's exports — `CommandBus`, `EventBus`, `QueryBus`, `EventPublisher` — are available to every module in the application without any of them needing to import `CqrsModule` explicitly. `HeroesGameModule` simply declares its handlers and sagas as providers, and NestJS's dependency injection resolves the bus and publisher dependencies from the global scope automatically.

The scoped variant in `ScopedModule` introduces two differences. The first is `Scope.REQUEST` in the `@CommandHandler` decorator options. This tells NestJS not to reuse a singleton handler instance across requests. Instead, it creates a brand new handler instance for each incoming request and tears it down when the request completes. This is necessary when the handler needs to carry state that is specific to one request, such as a caller's identity or a tracing context, without that state leaking into other concurrent requests.

The second difference is `AsyncContext`. Each scoped handler injects `@Inject(REQUEST) public readonly context: AsyncContext` and passes `this.context` as a second argument to `mergeObjectContext()`. `AsyncContext` is a container provided by `@nestjs/cqrs` that holds per-request data. Passing it to `mergeObjectContext()` ensures that when the hero commits events, those events are tagged with the current request's context. Downstream scoped event handlers that receive those events can then inject the same `AsyncContext` and read the per-request data from it, maintaining request isolation across the entire event chain.

The scoped saga's `dragonKilled` adds one more step that the regular saga does not have: `AsyncContext.merge(event, command)`. When the saga receives a `ScopedHeroKilledDragonEvent`, that event carries the `AsyncContext` from the request that originally triggered it. The new `ScopedDropAncientItemCommand` starts with no context of its own. `AsyncContext.merge` copies the context from the event onto the command before it is returned to the observable stream. When `CqrsModule` dispatches that command, the `ScopedDropAncientItemHandler` receives it and its own injected `AsyncContext` is populated correctly, maintaining the same request context all the way from the original command through the saga and into the downstream handler. Without this merge call, the context would be lost at the saga boundary and the second handler would operate in isolation with no awareness of the original request.
